BDD work flow

  1. Collaborate with all stakeholders on a Specification.
  2. Ensure understanding through examples written in light-weight mark-up e.g. Gherkin.
  3. Make Executable Specifications by parsing the mark-up.
  4. Watch these new Pending unimplemented tests fail!
  5. Implement the tests, watching them fail.
  6. Write the Implementation, watch it pass!

Isn't that just Test Driven Development?

Yep!

BDD is just TDD modified to include everyone not just the developer

It facilitates collaboration and empowers stake holders by including then.

It focuses on the behaviour rather than the implementation which allows for out of the box thinking and innovation.

It produces a body of living documentation that by design can not become outdated.

Gherkin

Is a Business Readable, ubiquitous testing Domain Specific Language.

Gherkin is a line-oriented language that uses indentation to define structure.

Gherkin consists of the following key words:

  • feature
  • background
  • scenario
  • scenario_outline
  • examples
  • given
  • when
  • then
  • and
  • but

And that's all!

An example of a Gherkin feature file

Feature: Serve coffee As a Software Developer I want a cup of coffee So that I can stay alert during a grooming session Scenario: Get a coffee Given there is at least 1 coffee pod And it is not decaf When I insert the pod into the machine And press the coffee button Then I should be served a coffee

How to Misuse Cucumber

In My Opinion

Misuse #1: Cucumber test written in isolation
If you aren't talking with your domain experts and stake holder when you create your Feature files, you're probably doing it wrong. If you aren't at least showing them your Feature files and making changes based on their feedback, you're definitely doing it wrong. If you aren't using Cucumber to enhance communication and collaboration, you're just writing automated tests and missing the whole point.
Misuse #2: Cucumber for Behaviour-After Development (BAD)
If you are writing your Feature files after implementation development you are by passing your stake holders and risk delivering in correct behaviour
Misuse #3: Cucumber as exclusively X test
Cucumber translates business requirements expressed as examples into Acceptance test Acceptance test should be written targeting the layer that most effectively exposes that requirement
Misuse #4: Premature optimization of Cucumber tests
Cucumber step definitions written with some care can enable significant re-use and composition resulting in rapid test development This can results in a temptation to over-engineer Cucumber Tests
