BDD

An introduction to Behaviour Driven Development

Need Assessment

  • Deficent requirements gathering processes
    • Stories being created at sprint planning
    • Grooming meetings lacking in focus
    • Variation in understood of requirements entering sprint
  • Deficent acceptance testing processes
    • Testers not aware of what needs to be tested
    • No formalized explorative testing strategy
    • Lacking any form of automation
  • Deficent Documentation
    • New features not being documented
    • Existing documentation becoming out of date
    • Documentation not tied to release artifact

What is Behaviour Driven Development

Using example
Real life concrete examples
at multiple levels
at all test levels acceptance integration and unit
to create a shared understanding
between all stake holders: product owner, BA, DBA, UX Designer, QA Tester, DevOps, Technical Writer, Marketing and Support
and surface uncertainty
identify and resolve mis-understanding before implementation
to deliver software that matters
so that the implementation delivered fulfils the end users needs

That sounds nice but how does that fullfil our needs

BDD work flow

  1. Collaborate with all stakehoders on a Specification.
  2. Ensure understanding through examples written in light weight markup eg. Gherkin.
  3. Make Executable Specifications by parsing the markup.
  4. Watch these new Pending unimplemented tests fail!
  5. Implement the tests, watching them fail.
  6. Write the Implementation, watch it pass!

Isn't that just Test Driven Development

Yep!

BDD is just TDD modified to include everyone not just the developer

It facilitates collaboration and empowers stake holders by including then.

It focuses on the behaviour rather than the implementation which allows for out of the box thinking and inovation.

It produces a body of living documentation that by design can not become outdated.

Gherkin

Is a Business Readable, ubiquitous testing Domain Specific Language

Gherkin is a line-oriented language that uses indentation to define structure.

Gherkin consists of the following key words:

  • feature
  • background
  • scenario
  • scenario_outline
  • examples
  • given
  • when
  • then
  • and
  • but

And thats all!

An example of a Gherkin feature file

Feature: Serve coffee As a Software Developer I want a cup of coffee So that I can stay awake to support a late night deployment Scenario: Get a coffee Given there is at least 1 coffee pod And it is not decaf When I insert the pod into the machine And press the coffee button Then I should be served a coffee
